# 07 - 测试与发布流程

> 本文档描述慧眼 SmartEye 的测试策略、TDD 实践、版本管理规范、双仓库同步、以及完整的发布流程。

---

## 1. 测试策略

### 1.1 测试金字塔

```
          ╱╲
         ╱  ╲         手动真机测试（安装、无障碍走查、闭眼测试）
        ╱ E2E╲        每个版本发布前必须
       ╱──────╲
      ╱ Widget ╲      Widget 测试（MVP 暂缓）
     ╱──────────╲
    ╱   单元测试    ╲   所有 Service 和 Model 必须覆盖
   ╱────────────────╲
```

当前 MVP 阶段以**单元测试**为核心，配合**手动真机测试**。Widget 测试和集成测试在功能稳定后补充。

### 1.2 TDD 铁律

> **NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST**

所有功能代码必须遵循 RED → GREEN → REFACTOR 流程：

1. **RED**：先写测试，运行确认失败
2. **GREEN**：写最少的代码让测试通过
3. **REFACTOR**：重构代码，保持测试通过

### 1.3 单元测试覆盖要求

| 模块类型 | 必须覆盖 | 说明 |
|---------|---------|------|
| Service 类 | 100% 公开方法 | 每个 public 方法必须有测试 |
| Model 类 | 100% | 包括序列化、反序列化、边界值 |
| Util 工具类 | 100% | 包括边界条件、异常输入 |
| Widget | 暂缓 | MVP 阶段不强制 |

### 1.4 一键测试脚本

项目提供了 `scripts/test.sh` 一键运行全部检查：

```bash
./scripts/test.sh
```

执行顺序：
1. `flutter analyze` — 静态分析，必须零警告
2. `dart format --output=none --set-exit-if-changed` — 格式检查
3. `flutter test test/unit/` — 单元测试
4. 集成测试（如果存在）

**提交前必须通过一键测试。**

## 2. 单元测试编写规范

### 2.1 测试文件位置

```
test/unit/
├── services/
│   ├── ocr_service_test.dart
│   ├── tts_service_test.dart
│   ├── history_service_test.dart
│   └── ...
└── models/
    ├── meal_code_test.dart
    └── ...
```

### 2.2 测试命名规范

```dart
group('OcrService', () {
  group('extractMealCodes', () {
    test('should extract code with # prefix', () { ... });
    test('should return empty list for no code', () { ... });
    test('should handle full-width # character', () { ... });
  });
});
```

- `group` 描述类名/方法名
- `test` 描述期望行为，使用 "should ... when/for ..." 格式
- 一个测试只测一个行为

### 2.3 测试原则

- **尽量使用真实代码，少用 mock**：Service 层测试使用真实实现，只 mock 外部依赖（如 MethodChannel）
- **测试边界条件**：空字符串、null、超长文本、特殊字符
- **测试异常路径**：确保错误场景也被覆盖
- **不要测试 Flutter 框架本身**：如 `setState` 是否触发重建，这是框架的责任
- **每个测试独立**：测试之间不共享状态，不依赖执行顺序

### 2.4 MethodChannel 测试

对于依赖 MethodChannel 的服务（如 AudioService、PermissionService），使用 Flutter 提供的 `TestDefaultBinaryMessenger`：

```dart
TestDefaultBinaryMessengerBinding.instance.defaultBinaryMessenger
    .setMockMethodCallHandler(_channel, (call) async {
  // 模拟原生端返回
  if (call.method == 'playAssets') return true;
  return null;
});
```

## 3. 手动测试检查清单

每次发布前必须在真机上完成以下检查：

### 3.1 核心功能测试

- [ ] 首次安装启动能听到完整教程
- [ ] 对准 #数字 小票能正确播报取餐码
- [ ] 对准美团/饿了么/京东外卖小票能识别平台名
- [ ] 倒置小票也能识别（180°双方向）
- [ ] 单击屏幕重听上一次结果
- [ ] 三击屏幕重新识别
- [ ] 上滑播报历史记录
- [ ] 下滑播报操作帮助
- [ ] 没有识别到小票时播报"没有识别到取餐码"
- [ ] 距离反馈音节奏正确（慢→快）
- [ ] 位置引导准确（请向左/右/上/下偏）

### 3.2 异常路径测试（NSF 红线）

- [ ] 拒绝摄像头权限时有语音提示
- [ ] 永久拒绝权限时语音提示并跳转设置
- [ ] 光线不足时有语音提示，手电自动开启
- [ ] 相机初始化失败时有语音提示
- [ ] 低亮度下手电不支持时有语音提示
- [ ] 下载更新失败时有语音提示
- [ ] 安装权限被拒时有语音提示
- [ ] **以上所有异常都不是静默失败**

### 3.3 无障碍测试（最重要）

- [ ] **闭眼测试**：闭上眼睛能完成取餐码识别全流程
- [ ] TalkBack 开启时所有功能正常
- [ ] 所有可交互元素有正确的 Semantics label
- [ ] 没有音频重叠现象
- [ ] 语音音量开到最大不破音
- [ ] 2 倍速下语音可懂

### 3.4 安全测试

- [ ] Release APK 用 release keystore 签名（非 debug）
- [ ] Release APK 无 logcat 输出
- [ ] 屏幕无调试覆盖层
- [ ] 不允许明文 HTTP 流量
- [ ] 临时文件使用后被清理
- [ ] 历史记录非明文存储

## 4. 版本管理规范

### 4.1 版本号格式

采用 Semantic Versioning：`MAJOR.MINOR.PATCH+BUILD`

- `MAJOR`：重大架构变更或正式发布（当前为 0，表示 MVP 阶段）
- `MINOR`：新增功能
- `PATCH`：Bug 修复、安全加固
- `BUILD`：构建号，每次构建递增

示例：`0.9.0+21` → v0.9.0 第 21 次构建

### 4.2 必须同步更新的文件

每次发版必须同时更新以下文件：

| 文件 | 更新内容 |
|------|---------|
| `pubspec.yaml` | `version: X.Y.Z+N` |
| `CHANGELOG.md` | 添加新版本条目，按"改了什么+为什么+影响"三要素 |
| `VERSION.md` | 添加版本历史表格行 |
| `docs/CHANGELOG.md` | 同步更新（归档用） |

### 4.3 CHANGELOG.md 规范

```markdown
## [0.9.0] — 2026-07-11 (安全加固发布候选版)

### 安全修复
- 修复 CVE-STYLE-014：release模式禁用debugPrint输出到logcat，防止OCR内容泄露
- 修复 CVE-STYLE-015：Android包名从com.example.smart_eye改为com.smart_eye，正式标识产品
- ...

### 改进
- 锁定所有pubspec直接依赖版本，防止意外升级
- 修复TtsService音频播放竞态条件，使用互斥锁防止语音重叠
```

- 版本章节严格**从新到旧降序排列**
- 每条变更包含「改了什么+为什么改+影响是什么」
- 无口语化表达、无占位符、无随意批注
- 分类标签：`安全修复` / `新功能` / `改进` / `Bug修复`

### 4.4 Git 标签规范

标签格式：`vX.Y.Z`（带 `v` 前缀）

```bash
git tag v0.9.0-rc1
git push origin v0.9.0-rc1
git push github v0.9.0-rc1
```

标签必须与 `pubspec.yaml` 中的版本号对应。

### 4.5 分支策略

- **main**：唯一主分支，始终保持可发布状态
- **feature/xxx**：功能分支，从 main 创建，合并回 main
- **禁止直接在 main 上开发**

使用 git worktree 进行功能开发：
```bash
# 创建功能分支和独立工作目录
git worktree add ../smart_eye_feature_xxx -b feat/xxx main

# 在新目录中开发
cd ../smart_eye_feature_xxx
flutter pub get
# ... 开发、测试 ...

# 合并回 main
cd ../smart_eye
git fetch . feat/xxx
git merge feat/xxx --ff-only
git worktree remove ../smart_eye_feature_xxx
```

## 5. 双仓库同步

### 5.1 远程仓库配置

```bash
git remote -v
# origin -> https://gitee.com/free-style_2_0/smart_eye.git （国内主仓库）
# github -> https://github.com/pomelo56/smart_eye.git     （海外镜像/比赛仓库）
```

### 5.2 同步规则

**每次推送必须同时推送到两个仓库：**

```bash
git push origin main
git push github main
git push origin --tags
git push github --tags
```

这不是可选步骤！Gitee 是国内用户的主要下载源（速度快），GitHub 是海外镜像和比赛提交仓库。

### 5.3 Release 同步

每次发版必须**同时创建 Gitee Release 和 GitHub Release**，两个 Release 都要上传 APK 文件：

```bash
# 发布到 Gitee
export GITEE_TOKEN=<你的令牌>
./scripts/release-gitee.sh v0.9.0-rc1 build/app/outputs/flutter-apk/app-release.apk

# 发布到 GitHub（需要 gh CLI 已登录）
./scripts/release-github.sh v0.9.0-rc1 build/app/outputs/flutter-apk/app-release.apk
```

**为什么必须两个都发？** 应用内更新逻辑是先尝试 Gitee（国内快），失败了 fallback 到 GitHub。如果只发了 Gitee Release，海外用户无法下载；只发了 GitHub Release，国内用户下载可能超时。

## 6. 完整发布流程

### 6.1 标准发布（推荐使用脚本）

```bash
# 确保在 main 分支且工作目录干净
git checkout main
git status  # 应该显示 "nothing to commit, working tree clean"

# 1. 运行一键测试
./scripts/test.sh

# 2. 构建 Release APK
flutter build apk --release

# 3. 安装到真机测试（可选但强烈推荐）
adb install -r build/app/outputs/flutter-apk/app-release.apk
# 在手机上完成手动测试检查清单

# 4. 更新版本号和 CHANGELOG
# 编辑 pubspec.yaml → version: X.Y.Z+N
# 编辑 CHANGELOG.md → 添加新版本条目
# 编辑 VERSION.md → 添加版本行

# 5. 提交版本变更
git add pubspec.yaml CHANGELOG.md VERSION.md docs/CHANGELOG.md
git commit -m "chore(release): bump version to vX.Y.Z"

# 6. 打标签
git tag vX.Y.Z

# 7. 推送到双仓库
git push origin main
git push github main
git push origin vX.Y.Z
git push github vX.Y.Z

# 8. 创建 Release 并上传 APK
./scripts/release-gitee.sh vX.Y.Z build/app/outputs/flutter-apk/app-release.apk
./scripts/release-github.sh vX.Y.Z build/app/outputs/flutter-apk/app-release.apk
```

### 6.2 使用 release.sh 一键发布

项目提供了 `scripts/release.sh` 自动化上述流程：

```bash
# 标准发布
./scripts/release.sh v0.9.0

# 发布并安装到设备
RELEASE_INSTALL=1 ./scripts/release.sh v0.9.0

# 发布并推送到双仓库 + 创建 Release
export GITEE_TOKEN=<你的令牌>
RELEASE_INSTALL=1 ./scripts/release.sh v0.9.0
```

`release.sh` 自动执行：
1. 检查在 main 分支且工作目录干净
2. 运行 `./scripts/test.sh`
3. `flutter build apk --release`
4. 推送到 origin 和 github
5. 推送标签
6. （可选）安装到 adb 设备
7. （可选）调用 release-gitee.sh 和 release-github.sh 创建 Release

## 7. 令牌配置

### 7.1 Gitee 令牌配置

1. 打开 https://gitee.com/personal_access_tokens
2. 新建令牌，勾选 `projects` 和 `releases` 两个权限
3. 生成令牌后，持久化到 `~/.zshrc`：
   ```bash
   echo 'export GITEE_TOKEN=你的令牌' >> ~/.zshrc
   source ~/.zshrc
   ```
4. **绝对不要**将令牌写入项目内的任何文件

### 7.2 GitHub 认证

使用 `gh` CLI 比直接使用 token 更安全：

```bash
# 安装 gh
brew install gh

# 浏览器登录（推荐）
gh auth login --web

# 验证
gh auth status
```

`gh` 的认证信息存储在 `~/.config/gh/hosts.yml`（权限 600），不会被 commit。

## 8. 无线调试部署

开发阶段频繁安装 APK 到真机，使用无线 ADB 无需数据线：

详细步骤参见 [`无线调试安装APK.md`](../无线调试安装APK.md)，简要流程：

```bash
# 1. 手机开启开发者选项 → 无线调试
# 2. 手机和电脑在同一 WiFi
# 3. 手机点"使用配对码配对设备"
adb pair <手机IP>:<配对端口>
# 输入配对码

# 4. 连接
adb connect <手机IP>:<连接端口>

# 5. 一键构建+安装
DEVICE=$(adb devices | awk 'NR==2 {print $1}')
flutter build apk --release && adb -s "$DEVICE" install -r build/app/outputs/flutter-apk/app-release.apk

# 6. 查看日志
adb -s "$DEVICE" logcat -s "SmartEye:D" "*:S"
```

**注意**：IP 和端口每次开启无线调试都会变化，不要写死在脚本中。

## 9. 版本回滚

如果发布后发现严重问题，按以下步骤回滚：

```bash
# 1. 找到上一个稳定版本的标签
git tag -l  # 列出所有标签

# 2. 硬重置到上一个稳定版本
git reset --hard v0.8.6  # 举例：回滚到 v0.8.6

# 3. 强制推送到双仓库（注意：会覆盖历史，仅在严重问题时使用）
git push origin main --force
git push github main --force

# 4. 删除有问题的标签
git tag -d v0.9.0-rc1
git push origin :refs/tags/v0.9.0-rc1
git push github :refs/tags/v0.9.0-rc1

# 5. 在 Gitee/GitHub 网页上删除对应的 Release
# 6. 重新构建上一个版本的 APK 并发布
```

## 10. 提交前自检清单

每次提交代码前，逐条勾选：

### 代码质量
- [ ] `./scripts/test.sh` 全部通过
- [ ] 新增代码有对应的单元测试，且先失败过（RED→GREEN）
- [ ] 没有引入新的第三方依赖（或已在 PR 说明中解释理由）
- [ ] 没有调用任何云端 OCR API

### 无障碍与语音（NSF 红线）
- [ ] 所有交互元素有 Semantics 包裹
- [ ] 所有用户操作有语音反馈
- [ ] 所有 `catch` 块和失败分支有语音播报（无静默失败）
- [ ] 新增异常场景的语音已录制并注册到 TtsService
- [ ] 没有音频重叠

### 隐私与安全
- [ ] 没有上传原图或相机帧到任何服务器
- [ ] 没有收集设备信息/位置信息
- [ ] release 模式无敏感日志输出
- [ ] 临时文件在 finally 中清理

### 文档与版本
- [ ] `CHANGELOG.md` 已更新
- [ ] `pubspec.yaml` 版本号已更新
- [ ] 如修改了核心流程，相关 Wiki 已更新
- [ ] 代码已格式化（`dart format lib/ test/`）
