# 02 - 架构设计详解

> 本文档详解慧眼 SmartEye 的整体架构、核心模块设计、数据流、以及关键架构决策背后的原因。

## 1. 整体架构

慧眼 SmartEye 是一个 **Flutter + Android 原生混合架构** 的应用：

- **Flutter/Dart 层**：UI、相机控制、OCR 识别、业务逻辑、手势处理、状态管理
- **Android/Kotlin 层**（MainActivity.kt）：提供 4 个 MethodChannel，处理音频播放、权限请求、APK 安装、签名验证等原生能力
- **端侧 AI**：Google ML Kit 文字识别（纯本地，无网络依赖）

## 2. 核心模块职责

### 2.1 HomeScreen — 唯一主页面

`lib/screens/home_screen.dart` 是应用唯一的页面，承担了几乎所有业务逻辑：

| 职责 | 说明 |
|------|------|
| 相机管理 | 初始化（3次重试）、预览、拍照、手电控制、生命周期绑定 |
| 扫描循环 | 每 2 秒拍一张照片，送入 OCR 处理 |
| OCR 管线 | 正向 + 180°旋转双方向识别、取餐码提取、平台检测、位置计算 |
| 多帧验证 | 3秒窗口跨帧合并码缓存，连续确认才播报 |
| 手势处理 | 单击重听、双击导出日志、三击重扫、上滑历史、下滑帮助 |
| 语音调度 | 所有场景的语音播报入口，NSF 红线执行点 |
| 更新流程 | 检查更新、确认下载、进度播报、安装 |
| 亮度检测 | 低亮度提示 + 自动开手电 |
| 方向引导 | 根据取餐码位置播报"请向左/右/上/下偏" |

**为什么只有一个页面？**
- 视障用户不需要复杂的导航层级
- 全屏作为触摸表面，最大化手势操作区域
- 减少页面跳转带来的认知负担

### 2.2 TtsService — 语音反馈中枢

`lib/services/tts_service.dart` 是整个应用最关键的服务之一：

**核心设计**：
- 不依赖系统 TTS 引擎（国产 ROM 兼容性问题），而是将文本映射为预录音频片段序列
- 保持与 `flutter_tts` 相同的 API 接口（`speak()`/`stop()`），未来切换 TTS 方案时调用方无需修改
- 使用 `_playbackMutex` 互斥锁，每次 `speak()` 前先 `await stop()`，防止音频重叠（CVE-STYLE-018修复）

**文本→音频映射规则**（按优先级）：

| 优先级 | 匹配内容 | 映射结果 |
|--------|---------|---------|
| 1 | 包含"欢迎使用慧眼" | `tutorial.mp3`（完整教程） |
| 2 | 包含"没有识别到取餐码" | `none.mp3` |
| 3 | 内部距离反馈码 | `beep_slow.mp3` / `beep_fast.mp3` |
| 4 | 取餐码（如"#15"、"美团外卖65号"） | 平台前缀 + 逐位数字 + "号" |
| 5 | 其他固定提示 | 对应的 mp3 文件 |

### 2.3 AudioService — 原生音频通道封装

`lib/services/audio_service.dart` 封装了与原生 `MediaPlayer` 的 MethodChannel 通信：

- MethodChannel：`com.smart_eye/audio`
- `playAssets()` 通过 `Completer` 等待原生端 `onPlaybackComplete` 回调，**不使用延时估算**
- 30 秒安全超时防止原生回调丢失时永久阻塞
- 静态 `_handlerSet` 标志防止多实例重复注册 MethodCallHandler

### 2.4 OcrService — 取餐码提取与验证

`lib/services/ocr_service.dart` 负责从 OCR 文本中提取取餐码：

**关键算法**：
- **双模式提取**：严格模式（`#数字`）+ 模糊模式（允许空格，需附近有平台关键词）
- **平台检测**：在取餐码周围 8 字符窗口内查找平台关键词，防止小票页脚其他平台名干扰
- **最佳码选择**：位数最多的码优先（真实取餐码通常比订单号长）
- **冷却去重**：同一号码 5 秒内不重复播报

**支持平台**：美团外卖、美团闪购、饿了么、京东外卖、淘宝闪购、朴朴超市、永辉超市

### 2.5 HistoryService — 历史记录

`lib/services/history_service.dart`：
- 使用 `shared_preferences` 存储最近 5 条记录
- 24 小时自动过期清理
- XOR + Base64 混淆存储（CVE-STYLE-008），防止 adb 直接读取明文
- 5 秒内重复码自动去重

### 2.6 PermissionService — 权限处理

`lib/services/permission_service.dart`：
- 不依赖 `permission_handler` 包（省约 2MB 体积），直接通过 MethodChannel 调用 Android API
- 定义 `PermissionPlatform` 抽象接口，测试可注入 fake 实现
- 正确区分"从未询问过"和"永久拒绝"（通过 `noBackupFilesDir` 下的标志文件，而非 SharedPreferences）

### 2.7 DownloadService — 更新下载

`lib/services/download_service.dart`：
- 使用 Dio 下载 APK，支持进度回调
- 下载前二次校验 URL 属于可信域名白名单（CVE-STYLE-004）
- 临时文件使用随机文件名，在 finally 块中清理

### 2.8 MainActivity.kt — 原生入口

`android/app/src/main/kotlin/com/smart_eye/MainActivity.kt` 暴露 4 个 MethodChannel：

| Channel | 功能 |
|---------|------|
| `com.smart_eye/audio` | MediaPlayer 顺序播放、停止、状态查询、完成回调 |
| `com.smart_eye/permission` | 相机权限检查/请求/跳转设置 |
| `com.smart_eye/installer` | 安装未知应用权限检查/设置/APK安装 |
| `com.smart_eye/apk_verifier` | APK 签名验证（防供应链攻击） |

## 3. 核心数据流

### 3.1 取餐码识别与播报流程

```
每2秒Timer触发
    │
    ▼
_scanFrame()
    │
    ├── 拍照（CameraController.takePicture()）
    │
    ├── _maybeHandleLuminance()
    │   └── 亮度检测 → 过暗? → 播报+开手电
    │
    ├── 双方向OCR
    │   ├── 正向 InputImage → ML Kit processImage()
    │   └── 180°旋转图 → ML Kit processImage()
    │       └── 旋转块坐标映射回正立坐标系
    │
    ├── 合并两方向文本块
    │
    ├── 多pass取餐码匹配
    │   ├── Pass1: 精确匹配 #数字 格式
    │   ├── Pass2: 同block含 # 和数字
    │   └── Pass3: 仅数字兜底
    │
    ├── 平台检测（邻近度算法，8字符窗口）
    │
    ├── 位置计算（中心点→九宫格方位）
    │
    ├── 跨帧合并（3秒窗口 _codeCache）
    │   └── 按位置排序，解决倒置小票每帧只识别一个码的问题
    │
    ├── 冷却检查（同一码5秒内不重复）
    │
    ├── 距离反馈音（beep_slow/beep_fast）
    │
    └── 语音播报（TtsService.speak()）
        └── "美团外卖 65 号 请向右上方偏"
            │
            ├── _mapTextToAssets() 映射音频片段
            ├── AudioService.playAssets()
            │   └── MethodChannel → 原生 MediaPlayer 顺序播放
            └── 保存到 HistoryService
```

### 3.2 音频播放流程（MethodChannel 通信）

```
Dart 端                          Kotlin 端
─────────                        ─────────
TtsService.speak("井 1 5")
    │
    ├── 先 await _audioMutex（互斥锁）
    ├── await stop()
    │       └── MethodChannel.invokeMethod("stop")
    │               └── stopPlayback() → release MediaPlayer
    │
    ├── 映射为 [jing.mp3, num_1.mp3, num_5.mp3]
    │
    └── audioService.playAssets(paths)
            │
            ├── 创建 Completer
            ├── MethodChannel.invokeMethod("playAssets", {paths, volume})
            │       │
            │       ├── stopPlayback()（停止旧播放）
            │       ├── playNext() 递归播放
            │       │   ├── assets.openFd("flutter_assets/$path")  ← 路径陷阱!
            │       │   ├── MediaPlayer.create()
            │       │   ├── setAudioAttributes(ACCESSIBILITY, SPEECH)
            │       │   ├── playbackParams.speed = 2.0f（2倍速）
            │       │   └── setOnCompletionListener → playNext()
            │       └── 全部播完 → methodChannel.invokeMethod("onPlaybackComplete")
            │
            └── 等待 Completer.future（30秒超时）
                    └── 释放 _audioMutex
```

## 4. 关键架构决策

### 4.1 为什么放弃 flutter_tts？

**问题**：OPPO/ColorOS 等国产 ROM 不允许第三方应用绑定系统 TTS 引擎。即使系统 TTS 在设置中能正常工作，`flutter_tts` 和原生 `TextToSpeech.onInit` 均返回 ERROR。

**决策**：MVP 改用预录音频 + Android MediaPlayer。
- 优点：100% 兼容所有设备、无需联网、响应速度快、语音风格统一
- 缺点：只能播报预录内容，动态文本（如任意数字串）需要拼接数字音频
- 扩展方式：新增提示语音需录制音频文件并放入 `assets/audio/`

### 4.2 为什么是持续扫描而非手动触发？

盲人用户看不到画面，无法判断：
- 摄像头是否对准了小票
- 小票是否在画面中
- 光线是否充足
- 何时该按快门

持续扫描（每 2 秒一帧）+ 多帧验证 + 距离反馈音，用户只需将手机对准小票方向，系统自动识别并播报。

### 4.3 为什么用多帧验证？

取餐码通常只有 1-3 位数字，单次 OCR 误识别率较高（例如把 `#15` 识别为 `#1` 或 `#5`）。连续 2 次（可配置）识别到相同结果才播报，大幅降低误报率。

### 4.4 为什么相机逻辑在 HomeScreen 而非独立 Service？

早期设计中有 `camera_service.dart`，但实际开发发现相机的初始化、拍照、生命周期、手电控制与 UI 状态强耦合（如 `_isCameraReady`、`_isProcessing` 等标志位直接影响 Widget 构建），抽离为独立 Service 反而增加了大量胶水代码。当前将相机逻辑内联在 `_HomeScreenState` 中，代码更内聚。

### 4.5 为什么不使用 permission_handler 包？

`permission_handler` 包引入了约 2MB 的原生代码，而本项目只需要相机权限。直接通过 MethodChannel 调用 Android 权限 API，减少 APK 体积，同时避免了该包在不同 Android 版本上的兼容性问题。

### 4.6 为什么用 2.0 倍速播放？

视障用户日常使用屏幕阅读器（如 TalkBack）通常设置在 1.5x-3x 倍速，对快速语音有较强的接受能力。2.0 倍速在保证可懂度的前提下，缩短了语音反馈等待时间，提升交互效率。

### 4.7 为什么做双方向OCR？

外卖小票经常被倒置放置（用户或外卖员随意放置），如果只识别正向文字，倒置小票无法识别。对图像做 180°旋转后再识别一次，旋转块的坐标通过 `(x,y) → (W-x, H-y)` 映射回正立坐标系，位置播报仍然准确。

## 5. 状态管理

项目**没有使用 Provider/Riverpod/Bloc 等状态管理框架**，原因：
- 只有一个页面，状态复杂度低
- 状态主要集中在 `_HomeScreenState` 中
- 语音播报是副作用型操作，不适合用响应式状态管理
- 引入框架增加包体积和学习成本

使用 Flutter 原生的 `StatefulWidget` + `setState()` + `ValueNotifier` 即可满足需求。

## 6. 目录结构与依赖关系

```
lib/
├── main.dart                     # 入口：初始化、方向锁定、debugPrint禁用
├── screens/
│   └── home_screen.dart          # 主页面（依赖所有 services）
├── services/
│   ├── tts_service.dart          # 语音中枢（依赖 audio_service）
│   ├── audio_service.dart        # 音频通道（无Flutter业务依赖）
│   ├── ocr_service.dart          # OCR提取（纯Dart，无Flutter依赖）
│   ├── history_service.dart      # 历史记录（依赖 shared_preferences）
│   ├── download_service.dart     # 下载（依赖 dio + 安全检查）
│   ├── permission_service.dart   # 权限（MethodChannel抽象）
│   └── update_service.dart       # 更新（依赖 download + permission）
├── models/
│   ├── meal_code.dart            # 取餐码模型（纯数据）
│   └── update_info.dart          # 更新信息模型（纯数据）
└── utils/
    └── log_utils.dart            # 日志工具
```

**依赖方向**：`screens` → `services` → `models`/`utils`，不允许反向依赖。

## 7. 安全架构（v0.9.0 起）

参见 [06-安全加固实战总结](./06-安全加固实战总结.md)，核心安全层：

- **网络层**：禁止明文流量、证书校验、URL 白名单
- **安装层**：APK 签名验证、FileProvider 最小权限
- **存储层**：历史记录混淆存储、临时文件安全清理、noBackupFilesDir
- **日志层**：release 模式禁用 debugPrint、无堆栈输出、无敏感日志
- **构建层**：release 签名、ProGuard 混淆、ABI 限制(arm64-v8a)、依赖版本锁定
