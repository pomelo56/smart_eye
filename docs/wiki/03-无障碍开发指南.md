# 03 - 无障碍开发指南

> 本文档是慧眼 SmartEye 项目最核心的经验沉淀。视障应用开发与普通应用有本质区别，许多在普通应用中被忽略的细节，在视障应用中是**致命问题**。

---

## 1. 核心理念：NSF 红线（No Silent Failure）

> **视障工具绝不允许静默失败。**

这是本项目最高优先级的设计原则，高于一切功能需求和性能优化。

### 为什么？

普通应用出错时，用户可以：
- 看到红色错误提示
- 看到 Snackbar/Toast
- 看到按钮变色
- 截图求助

**视障用户什么都看不到。** 如果出错了没有声音，他们唯一能感知到的是「手机没反应了」，不知道发生了什么、该怎么办，只能强行重启应用。

### 什么是静默失败？

以下都是**禁止**的静默失败模式：

```dart
// ❌ 反面例子1：只打日志，不说话
try {
  await initializeCamera();
} catch (e) {
  print('Camera error: $e');  // 视障用户看不到日志！
}

// ❌ 反面例子2：只改UI状态
if (permissionDenied) {
  setState(() => _showError = true);  // 红色错误提示框，视障用户看不到！
}

// ❌ 反面例子3：只弹 Snackbar
ScaffoldMessenger.of(context).showSnackBar(
  SnackBar(content: Text('下载失败')),  // TalkBack可能不会读，且很快消失！
);

// ❌ 反面例子4：只改颜色
if (isError) {
  _buttonColor = Colors.red;  // 视障用户看不到颜色变化！
}
```

**正确做法**：所有异常路径都必须有语音播报，告诉用户发生了什么、下一步该怎么办。

```dart
// ✅ 正面例子：每个失败分支都有语音
if (status.isPermanentlyDenied) {
  await _ttsService.speakCameraPermissionPermanentlyDenied();
  // "请到系统设置开启摄像头权限"
  await Future.delayed(const Duration(seconds: 2));
  await _permissionService.openAppSettings();
}
```

## 2. 必须语音播报的场景清单

| 场景 | 必播内容 | 对应方法 |
|------|---------|---------|
| 摄像头权限被拒 | "未获得摄像头权限，请开启" | `speakCameraPermissionDenied()` |
| 摄像头权限永久拒绝 | "请到系统设置开启摄像头权限" → 跳转设置 | `speakCameraPermissionPermanentlyDenied()` |
| 手电硬件不支持 | "自动打开手电失败，请手动开启" | `speakTorchFailed()` |
| 相机初始化3次失败 | "摄像头不可用，请检查权限" | `_initCameraWithRetry()` 失败分支 |
| OCR 未识别到取餐码 | "没有识别到取餐码，请重新对准小票" | `prefix: none` |
| 光线不足 | "光线不足，请开灯或移到亮处" | `speakLuminanceDim()` |
| 下载更新失败 | "下载失败，请检查网络后重试" | `speakDownloadFailed()` |
| 安装权限被拒 | "请允许安装未知应用后重试" | `speakInstallPermissionDenied()` |
| 首次启动 | 完整操作教程（30秒左右） | `tutorial.mp3` |
| 识别到取餐码 | "XX外卖 XX号" | `speakMealCodeWithPlatform()` |
| 取餐码位置偏移 | "请向左/右/上/下偏" | 位置引导播报 |
| 上滑/下滑手势 | 历史记录 / 操作帮助 | `speakHistory()` / 帮助语音 |
| 有更新可用 | "发现新版本，上滑更新，下滑取消" | `speakUpdateAvailable()` |
| 下载中 | 进度播报（每20%） | `speakDownloading()` |

### 异常播报模板

```
"出问题了"（或具体原因）
"建议怎么办"（具体可操作步骤）
```

例如：
- ❌ "错误" → 用户不知道是什么错误，不知道怎么办
- ❌ "相机异常" → 什么异常？重启？重装？
- ✅ "摄像头不可用，请检查是否授予了摄像头权限" → 用户知道原因和操作步骤

## 3. Semantics（语义）规范

每个可交互 Widget 必须用 `Semantics` 包裹，且 `label` 必须描述「这是什么 + 操作后会怎样」。

```dart
// ❌ 反面例子：无 Semantics
GestureDetector(
  onTap: _replay,
  child: Container(color: Colors.black),  // 全屏黑色，TalkBack无法识别
);

// ✅ 正面例子：完整 Semantics
Semantics(
  label: '慧眼主界面。单击重听上一次结果，三击重新识别，上滑查看历史，下滑查看帮助。',
  button: true,
  enabled: true,
  onTap: _replay,
  onLongPress: _clearAndRescan,  // 如果有长按操作
  child: GestureDetector(
    onTap: _replay,
    onDoubleTap: _exportLogs,
    onLongPress: _clearAndRescan,  // 三击逻辑在内部实现
    child: CameraPreview(_cameraController!),
  ),
);
```

### Semantics label 编写原则

1. **不要描述外观**：❌ "黑色矩形" → ✅ "慧眼主界面"
2. **要描述功能**：❌ "按钮" → ✅ "单击重听，三击重新识别"
3. **要描述结果**：❌ "点击更新" → ✅ "点击后将下载并安装新版本"
4. **简洁明了**：不要写长句，TalkBack 朗读需要时间
5. **首次进入时朗读**：新页面/新状态首次出现时主动播报一次

### 注意事项

- `excludeSemantics: true` 只用于纯视觉装饰性 Widget（如背景图）
- 不要让两个有语义的 Widget 重叠，否则 TalkBack 会混淆
- 动态变化的内容用 `SemanticsService.announce()` 主动通知 TalkBack

```dart
// 状态变化时主动播报
SemanticsService.announce(
  '识别到美团外卖65号',
  TextDirection.ltr,
);
```

## 4. 手势设计原则

### 4.1 禁止复杂手势

| 禁止手势 | 原因 |
|---------|------|
| 双指缩放/旋转 | 视障用户无法确定双指位置关系 |
| 需要视觉定位的精确点击 | 视障用户看不到屏幕上的按钮位置 |
| 长按+滑动组合 | 学习成本高，容易误触 |
| 快速连续点击（>3次/秒） | 视障用户操作速度有限 |

### 4.2 推荐手势

本项目使用的手势方案：

| 手势 | 功能 | 设计理由 |
|------|------|---------|
| 单击 | 重听上一次结果 | 最简单、最不容易误触 |
| 三击 | 清除结果，重新识别 | 单击重听太频繁，三击不易误触 |
| 双击 | 导出诊断日志 | 调试功能，用户不常用 |
| 上滑 | 播报历史记录（或确认更新） | 上下滑是自然的浏览手势 |
| 下滑 | 播报操作帮助（或取消更新） | 与上滑对称 |

### 4.3 手势设计注意事项

1. **全屏幕作为触摸区域**：不要把按钮放在固定位置，整个屏幕都响应手势
2. **手势区分要有时间缓冲**：单击和三击需要用 Timer 区分（本项目用 600ms）
3. **每个手势操作都要有语音反馈**：用户操作后必须立刻听到声音确认
4. **避免手势冲突**：上下滑和点击不要同时触发
5. **手势容错**：三击的节奏宽容度要高，不要要求用户精确快速点击

```dart
// 手势实现模式
DateTime? _lastTapTime;
int _tapCount = 0;

void _handleTap() {
  _tapCount++;
  _lastTapTime = DateTime.now();
  
  Future.delayed(const Duration(milliseconds: 600), () {
    if (DateTime.now().difference(_lastTapTime!).inMilliseconds >= 600) {
      if (_tapCount == 1) {
        _replayLastResult();  // 单击
      } else if (_tapCount >= 3) {
        _clearAndRescan();    // 三击（或更多，容错）
      }
      _tapCount = 0;
    }
  });
}
```

## 5. 语音反馈设计

### 5.1 语音是唯一的信息通道

对于全盲用户，语音是他们获取信息的**唯一途径**。这意味着：

- 所有视觉上可见的信息，都必须有对应的语音表达
- 语音要简洁、清晰、快速
- 语音要先讲最重要的信息，再讲细节
- 同一时间只能有一个声音在播放（禁止音频重叠）

### 5.2 语音内容设计

**播报顺序**：核心结果 → 辅助信息 → 引导方向

```
✅ 好的例子："美团外卖 65 号 请向右上方偏"
   核心结果：美团外卖65号
   引导信息：向右上方偏

❌ 差的例子："请将手机向右上方移动，现在识别到了一个取餐码，是美团外卖的，号码是65号"
   语序混乱，引导信息在结果前面，用户需要听完才知道结果
```

**简洁原则**：
- 数字逐位朗读："15" 读 "一 五"，不读"十五"（避免与"50"等混淆）
- `#` 读"井"（取餐码标准读法）
- 平台名读全称："美团外卖"，不读"美团"
- 避免冗余词汇：❌"您的取餐码是" → ✅ 直接报号码

### 5.3 语音节奏控制

- 使用 2.0 倍速播放（视障用户习惯高速语音）
- 两个独立播报之间要有短暂停顿
- 紧急信息（如错误提示）打断当前播放，优先播报
- 非紧急信息（如距离反馈音）不打断主播报

### 5.4 音频互斥（CVE-STYLE-018 修复的教训）

早期版本中，连续调用 `speak()` 会导致音频重叠（前一句没说完后一句就开始），用户听到的是嘈杂的混合音。

**错误的修复方式**：用 `delay(Duration(seconds: 2))` 等语音播完。问题是音频时长不确定（数字长度不同、平台名长度不同）。

**正确的修复方式**：
- 使用互斥锁（Mutex）确保同时只有一个播放任务
- 每次新的 `speak()` 调用前，先 `stop()` 停止当前播放
- 底层通过 `MediaPlayer` 的 `onCompletion` 回调确认播放完成，不做时间估算

## 6. 距离反馈与引导（倒车雷达原理）

盲人用户无法看到手机是否对准了小票，需要声音反馈来调整位置：

### 6.1 反馈音设计

| 状态 | 反馈音 | 含义 |
|------|--------|------|
| 未检测到文字 | 慢"滴"声（每2秒一次） | 大范围搜索，移动手机 |
| 检测到文字但无法识别取餐码 | 快"滴"声（每0.5秒一次） | 接近了，微调位置 |
| 识别到取餐码 | 播报结果 | 对准了，保持不动 |

这模仿了倒车雷达的设计：越接近目标，声音频率越高。

### 6.2 位置引导

识别到取餐码后，根据取餐码在画面中的位置，播报调整方向：

| 取餐码位置 | 播报 |
|-----------|------|
| 画面中间 | 只播报号码（无引导） |
| 偏左/偏右/偏上/偏下 | "请向左偏"/"请向右偏"/"请向上偏"/"请向下偏" |
| 四个角 | "请向左上方偏" 等 |

**冷却机制**：位置引导 8 秒内不重复播报，避免信息轰炸。

### 6.3 双方向识别

小票经常被倒置放置，需要：
1. 对正向图像做 OCR
2. 对 180° 旋转图像做 OCR
3. 旋转后的坐标映射回正立坐标系：`(x, y) → (W - x, H - y)`
4. 合并两方向的识别结果，取置信度更高的

## 7. 首次启动体验

视障用户第一次打开应用时，**必须**听到完整的操作教程，因为：
- 他们看不到界面上的操作提示
- 他们不知道该怎么操作
- 应用商店的截图对他们无用

教程内容应覆盖：
1. 这是什么应用（"欢迎使用慧眼 SmartEye"）
2. 能做什么（"帮您识别外卖小票上的取餐码"）
3. 怎么用（"将手机对准小票，保持30厘米距离"）
4. 手势操作说明（"单击重听，三击重新识别，上滑历史，下滑帮助"）
5. 权限说明（"需要使用摄像头来识别小票"）

教程语音约 20-30 秒，使用 `shared_preferences` 标记是否已播放，仅首次启动播放。

## 8. 权限处理的无障碍注意事项

### 8.1 权限请求前要语音说明

弹出系统权限对话框前，必须先语音告诉用户即将发生什么：

```dart
// ✅ 正确
await _ttsService.speak("慧眼需要使用摄像头来识别小票，请在弹出的对话框中选择允许");
await Future.delayed(const Duration(seconds: 2));
final status = await _permissionService.requestCameraPermission();
```

如果不提前说明，用户突然听到系统权限对话框的 TalkBack 提示，不知道是什么应用、为什么要权限。

### 8.2 永久拒绝的特殊处理

当用户选择"不再询问"后：
1. 语音告知："请到系统设置开启摄像头权限"
2. 等待 2 秒让用户听完
3. 自动跳转到应用设置页
4. 不要反复弹出系统对话框（已经没用了）

### 8.3 权限标志存储的陷阱

不要用 `SharedPreferences` 存储「是否请求过权限」标志！因为 ColorOS/Google 备份会在卸载重装后恢复 SharedPreferences，导致首次安装就被误判为"永久拒绝"。

正确做法：在 `noBackupFilesDir` 下创建一个标志文件，这个目录不会被备份恢复。

## 9. 测试无障碍功能

### 9.1 必须闭眼测试

开发视障应用时，**闭着眼睛测试**是最基本的要求：
1. 闭眼安装应用
2. 闭眼完成首次启动
3. 闭眼对准一张小票
4. 闭眼完成重听、重新识别、查看历史等操作
5. 闭眼处理权限拒绝场景

如果闭眼能完成所有核心流程，无障碍才算合格。

### 9.2 开启 TalkBack 测试

在 Android 设备上开启 TalkBack（设置 → 无障碍 → TalkBack），测试：
- 所有可交互元素是否有正确的语义标签
- 焦点移动是否合理
- 动态内容是否被正确朗读
- 手势操作是否被 TalkBack 拦截

### 9.3 手动测试检查清单

- [ ] 首次启动能听到完整教程
- [ ] 所有按钮/可交互区域有 Semantics label
- [ ] 权限被拒时有语音说明
- [ ] 相机初始化失败时有语音说明
- [ ] 识别成功时有语音播报
- [ ] 识别失败时有语音提示
- [ ] 单击/三击/上滑/下滑都有语音反馈
- [ ] 没有语音重叠现象
- [ ] 低亮度时有提示
- [ ] 位置引导准确
- [ ] 闭眼能完成核心取餐码识别流程
- [ ] TalkBack 开启时所有功能正常

## 10. 常见无障碍错误

| 错误 | 后果 | 修复 |
|------|------|------|
| 异常只打日志不说话 | 用户以为手机卡死了 | 所有 catch 分支加语音 |
| 只用颜色表示状态 | 色盲/全盲用户看不到 | 加语音/声音反馈 |
| 手势区域太小 | 视障用户点不到 | 全屏响应手势 |
| 语音太快/太慢 | 听不清或等待太久 | 2.0倍速，清晰吐字 |
| 同时播放多个声音 | 听不清内容 | 互斥锁+先stop再play |
| 新内容出现不主动播报 | 用户不知道有新内容 | SemanticsService.announce() |
| 弹窗/对话框无语音提示 | 用户不知道弹窗出现了 | 弹窗出现前先语音预告 |
| 按钮只有图标没有文字 | TalkBack读"未标记按钮" | 加 Semantics label |

## 11. 其他视障项目可复用的经验

1. **音频优先架构**：将语音反馈作为一等公民，不是 UI 的附属品
2. **NSF 红线**：代码审查时检查所有 `catch` 和 `if (error)` 分支是否有语音
3. **预录音频方案**：在系统 TTS 不可靠时（国产 ROM），预录音频是最稳妥的方案
4. **持续扫描模式**：不要让用户主动拍照，系统自动扫描+多帧验证
5. **倒车雷达式反馈**：用声音频率表示距离/状态，用户不需要看屏幕
6. **全屏幕手势**：不要放视觉按钮，整个屏幕都是触摸区域
7. **闭眼测试**：这是最简单有效的无障碍测试方法
8. **2倍速语音**：视障用户的信息接收速度远快于明眼人
