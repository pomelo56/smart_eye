# 06 - 安全加固实战总结

> 本文档总结慧眼 SmartEye v0.8.4 ~ v0.9.0-rc1 期间完成的 18 个安全漏洞修复经验。包括漏洞分类、修复方案、Android 安全配置最佳实践，可供其他移动端项目参考。

---

## 1. 安全审计概述

v0.8.3 版本进行了一次完整的安全审计（扮演黑客攻击视角），共发现 18 个安全漏洞，按严重程度分为四级：

| 级别 | 数量 | 标识 | 修复版本 |
|------|------|------|---------|
| 严重 (Critical) | 2 | CVE-STYLE-001~002 | v0.8.4 |
| 高危 (High) | 4 | CVE-STYLE-004~007 | v0.8.5 |
| 中危 (Medium) | 6 | CVE-STYLE-008~013 | v0.8.6 |
| 低危 (Low) | 6 | CVE-STYLE-014~018 | v0.9.0-rc1 |

**数据收集审计结论**：慧眼 SmartEye **不收集任何用户个人信息**：
- 不收集设备信息、位置信息
- 不上传照片或相机帧
- 不使用云端 OCR 或分析服务
- 历史记录仅在本地加密存储

## 2. 严重漏洞修复

### CVE-STYLE-001：APK 更新无签名验证（供应链攻击）

**漏洞**：应用内更新下载 APK 后直接安装，没有验证 APK 签名是否与当前应用一致。攻击者可以通过 DNS 劫持或恶意热点，将更新请求重定向到恶意 APK，用户安装后应用被替换。

**修复**：在原生层实现 APK 签名验证：
```kotlin
// MainActivity.kt - verifyApkSignature()
// 1. 获取当前应用的签名证书指纹
// 2. 解析下载 APK 的签名证书
// 3. 比对两者是否一致，不一致则终止安装
```

**经验**：
- Android 13+ 使用 `GET_SIGNING_CERTIFICATES`，旧版本用 `GET_SIGNATURES`
- 必须在原生层做验证，Dart 层的校验可以被 Frida/Xposed hook 绕过
- 使用 FileProvider 授予 URI 权限，不申请存储权限

### CVE-STYLE-002：Release APK 使用 debug 签名

**漏洞**：没有配置 release signingConfig，Gradle 默认用 debug keystore 签名 release 构建。debug keystore 是公开已知的，攻击者可以用相同密钥签名恶意 APK 覆盖安装。

**修复**：
1. 生成专用 release keystore
2. 创建 `key.properties`（不提交到 git）
3. 配置 `build.gradle` 的 signingConfigs

```gradle
// build.gradle
signingConfigs {
    release {
        keyAlias keystoreProperties['keyAlias']
        keyPassword keystoreProperties['keyPassword']
        storeFile file(keystoreProperties['storeFile'])
        storePassword keystoreProperties['storePassword']
    }
}
buildTypes {
    release {
        signingConfig signingConfigs.release
        minifyEnabled true  // 开启 ProGuard 混淆
        shrinkResources true
    }
}
```

## 3. 高危漏洞修复

### CVE-STYLE-004：无 URL 白名单校验（中间人+恶意下载）

**漏洞**：更新下载 URL 可以是任意地址，没有域名白名单限制。

**修复**：实现 URL 白名单校验：
```dart
// UpdateService.isValidDownloadUrl()
static const _trustedDomains = [
  'gitee.com',
  'github.com',
  // 其他可信域名
];

static bool isValidDownloadUrl(String url) {
  final uri = Uri.tryParse(url);
  if (uri == null || (uri.scheme != 'https' && !kDebugMode)) return false;
  return _trustedDomains.any((domain) => uri.host.endsWith(domain));
}
```

**双层校验**：
- Dart 层：发起下载前校验
- Kotlin 层：安装前可做二次校验（纵深防御）

### CVE-STYLE-005：明文 HTTP 流量允许

**漏洞**：没有配置 network_security_config，Android 9+ 虽然默认禁止明文，但 debug 覆盖可能允许。

**修复**：创建 `res/xml/network_security_config.xml`：
```xml
<network-security-config>
    <base-config cleartextTrafficPermitted="false">
        <trust-anchors>
            <certificates src="system" />
        </trust-anchors>
    </base-config>
    <!-- debug 模式允许本地明文 -->
    <debug-overrides>
        <trust-anchors>
            <certificates src="user" />
        </trust-anchors>
    </debug-overrides>
</network-security-config>
```

在 AndroidManifest.xml 中引用：
```xml
<application android:networkSecurityConfig="@xml/network_security_config">
```

### CVE-STYLE-006：日志中输出敏感信息

**漏洞**：OCR 识别结果（小票内容可能包含姓名、电话、地址）通过 `debugPrint` 输出到 logcat，任何有 `READ_LOGS` 权限的应用或连接 adb 的电脑都能读取。

**修复**：
- release 模式下禁用 `debugPrint`（覆盖为空函数）
- 所有日志使用统一的 `LogUtils`，release 模式自动过滤
- 禁止在日志中输出完整 OCR 文本

### CVE-STYLE-007：外部存储日志可控

**漏洞**：日志导出功能默认开启，将详细日志写入外部存储。

**修复**：
- 日志导出改为隐藏手势触发（双击屏幕）
- 写入前检查存储可用性
- 日志文件不含敏感 OCR 内容

## 4. 中危漏洞修复

### CVE-STYLE-008：本地存储数据明文

**漏洞**：历史记录（取餐码）通过 SharedPreferences 明文存储，root 设备或 adb 可以直接读取。

**修复**：XOR + Base64 混淆存储：
```dart
// 注意：XOR 不是强加密，仅阻止随意 grep
// 强加密应使用 flutter_secure_storage（Android Keystore）
String _obfuscate(String plain) {
  final key = 'SmartEyev0.8.6';
  final bytes = utf8.encode(plain);
  final xored = List<int>.generate(bytes.length, (i) =>
    bytes[i] ^ key.codeUnitAt(i % key.length));
  return base64.encode(xored);
}
```

**为什么不用 AES/Keystore**：MVP 阶段取餐码不属高度敏感数据，XOR 混淆已足够防止 casual 窥探。强加密增加代码复杂度和崩溃风险。

### CVE-STYLE-009：release 模式输出异常堆栈

**漏洞**：catch 块中 `print(e)` 输出完整堆栈到 logcat，暴露类名、方法名、文件路径。

**修复**：release 模式下仅在 `kDebugMode` 时输出详细堆栈。

### CVE-STYLE-010：临时文件可预测文件名

**漏洞**：拍照临时文件使用固定文件名（如 `temp_ocr.jpg`），攻击者可通过路径遍历或文件覆盖攻击。

**修复**：使用随机文件名：
```dart
final random = Random.secure();
final fileName = List.generate(16, (_) => random.nextInt(256))
    .map((b) => b.toRadixString(16).padLeft(2, '0'))
    .join();
final tempFile = File('${tempDir.path}/$fileName.jpg');
```

### CVE-STYLE-011：FileProvider 路径范围过大

**漏洞**：file_paths.xml 配置了根路径 `<external-path name="." path="."/>`，FileProvider 可以访问整个外部存储。

**修复**：严格限制到应用专用缓存目录：
```xml
<paths>
    <cache-path name="cache" path="." />
    <files-path name="files" path="." />
    <!-- 删除了 external-path 的根配置 -->
</paths>
```

### CVE-STYLE-012：临时文件未在 finally 中清理

**漏洞**：拍照临时文件在异常路径下未删除，长期使用后缓存目录堆积。

**修复**：使用 try-finally 确保删除：
```dart
try {
  await _processImage(tempFile.path);
} finally {
  if (await tempFile.exists()) await tempFile.delete();
}
```

### CVE-STYLE-013：屏幕调试覆盖层在 release 可见

**漏洞**：屏幕上半透明的调试日志面板（显示 OCR 结果）在 release 模式仍然显示，旁人可以从屏幕上看到识别到的内容。

**修复**：仅在 `kDebugMode` 时构建调试覆盖层。

## 5. 低危漏洞修复

### CVE-STYLE-014：release 模式 debugPrint 输出

**修复**：在 `main()` 中 release 模式下覆盖 debugPrint 为空函数。

### CVE-STYLE-015：包名使用 com.example 占位

**漏洞**：包名 `com.example.smart_eye` 是 Android Studio 自动生成的占位包名，可能与其他应用冲突，且不标识正式产品。

**修复**：
- 包名改为 `com.smart_eye`
- 修改 `build.gradle` 的 `applicationId` 和 `namespace`
- 移动 Kotlin 源文件到 `com/smart_eye/` 目录
- 更新 AndroidManifest.xml

**注意**：包名变更后，旧版本（com.example.smart_eye）无法覆盖安装，需先卸载。

### CVE-STYLE-016：应用描述为默认示例

**修复**：更新 `pubspec.yaml` 中的 `description` 为正式产品描述。

### CVE-STYLE-017：依赖版本使用 ^ 范围

**漏洞**：`pubspec.yaml` 中依赖使用 `^x.y.z` 版本范围，`flutter pub get` 可能自动更新到有破坏性变更或安全漏洞的新版本。

**修复**：锁定所有直接依赖为精确版本（去掉 `^`）：
```yaml
# 修复前
camera: ^0.10.0+1
# 修复后
camera: 0.10.5+9
```

**为什么不用 pubspec.lock**：lock 文件锁定的是传递依赖，直接依赖的 `^` 范围在 `pub upgrade` 时仍然会升级。显式指定精确版本更安全。

### CVE-STYLE-018：音频播放竞态条件

**漏洞**：连续调用 `speak()` 导致 MediaPlayer 资源竞争，可能崩溃或无声。

**修复**：在 TtsService 层添加互斥锁，每次 speak 前先 stop。详见 [04-音频系统实现](./04-音频系统实现.md)。

## 8. 安全审计方法论

本次安全审计采用了以下方法，其他项目可参考：

1. **攻击视角**：假设自己是攻击者，如何利用这个应用做恶意事情
2. **数据流跟踪**：从外部输入（网络、相机、文件）到内部处理到输出（存储、日志、网络），每个环节检查安全问题
3. **OWASP Mobile Top 10 对照**：逐项检查 M1-M10 是否存在问题
4. **权限审计**：检查每个申请的权限是否真正必要
5. **配置审计**：检查 Manifest、build.gradle、ProGuard、network_security_config 等配置文件
6. **代码审查**：grep 查找敏感关键词（`http://`、`print(`、`debugPrint`、`TODO`、临时文件操作、权限相关代码）
7. **依赖审计**：检查 pubspec.yaml 中依赖是否有已知漏洞（`flutter pub outdated`）

## 9. 给其他 Flutter 项目的安全建议

1. **从第一天起配置 release 签名**：不要等到发布前才处理
2. **创建 network_security_config.xml**：默认禁止明文流量
3. **使用 Flutter 自带的 `kReleaseMode`**：区分 debug 和 release 行为
4. **覆盖 debugPrint**：release 模式下禁用
5. **统一日志工具**：不要到处用 `print()`，封装一个 LogUtils
6. **文件操作永远用 try-finally**：确保临时文件被清理
7. **不要在日志中输出用户数据**：OCR 结果、位置信息、用户输入等
8. **锁定依赖版本**：不要在 pubspec.yaml 中使用 `^` 表示直接依赖
9. **配置 ProGuard**：release 构建开启混淆
10. **限制 ABI**：如果只面向国内市场和现代设备，只打 arm64-v8a 即可
11. **FileProvider 路径最小化**：不要配置根路径
12. **allowBackup=false**：除非确实需要云端备份
13. **NSF 原则也适用于安全错误**：安全相关的错误（如签名验证失败）也要告知用户（语音/对话框）
