## 阶段一：原型验证（2026-06，v0.1-v0.3）

### 核心目标
验证「用手机摄像头识别外卖取餐码并语音播报」这个想法是否可行。

### 技术选型与决策
- **初始方案**：Python + YOLO + PaddleOCR，跑在 Mac 上
- **问题暴露**：
  - Mac 端无法让视障用户随身携带使用
  - 必须做成手机 App
  - PaddleOCR 打包到手机端体积太大
- **决策**：迁移到 **Flutter + Android**，OCR 选用 **Google ML Kit 本地模型**（纯本地、不上传、隐私友好）

### 阶段成果
- 验证了取餐码识别的技术可行性
- 确定了 Flutter 技术栈
- 早期版本归档在 `docs/CHANGELOG.md`（已停止维护）

## 阶段二：MVP 核心功能（2026-07-04 ~ 2026-07-06，v0.4-v0.6.2）

### 核心目标
把产品从"能跑"做到"能用"，覆盖主流外卖平台，优化体验和体积。

### v0.4.0 — 多平台与多码支持（2026-07-04）
**里程碑**：从只支持美团单一平台，扩展到四大外卖平台。

**新增能力**：
- 支持美团、饿了么、京东、淘宝闪购四大平台
- 多码同时识别（一屏多个取餐码）
- 位置播报（左上/右上/左下/右下/中间）
- 平台名称语音播报（新增平台名音频素材）
- App 图标替换为自定义 logo
- 版本管理规范建立（`VERSION.md` + git tag）

**踩坑**：
- 取餐码匹配规则过松会误识别（如价格中的数字）
- 平台名与取餐码可能被 OCR 分到不同 TextBlock，需要近距离窗口匹配

### v0.5.0 — 音频播放可靠性（2026-07-04）
**里程碑**：解决音频时序问题，播放流程从"hack 等待"改为"原生回调驱动"。

**关键改进**：
- Kotlin 层播放完成后通过 MethodChannel 通知 Dart，`playAssets` 真正阻塞到播放完成
- 历史记录分条播放（「第1条 65号」「第2条 2号」）
- 移除所有 `Future.delayed` 时序 hack（之前靠"等几秒"来猜音频播完了，极不可靠）

**踩坑**：`AudioService._isInitialized` 硬编码为 `true`，导致初始化失败时静默无声 → 新增 `ping` 检测真实状态。

### v0.5.1 — 高优先级 Bug 修复（2026-07-05）
修复了多个影响使用的严重问题：
- 音频初始化真实检测
- `stop()` 返回 bool 让调用方感知是否成功
- 日志每次触发 `setState` 重建整棵 widget 树 → 改为 `ValueListenableBuilder`
- OCR 冷却 map 只增不减的内存泄漏
- MethodChannel handler 重复注册风险

### v0.6.0 — 「发现外卖」智能提示（2026-07-05）
**里程碑**：OCR 识别到平台关键词但还没找到取餐码时，提前给用户反馈。

**用户体验改进**：
- 识别到「美团」「饿了么」等关键词但未找到码时，播报「发现外卖，识别中，手机请稳一些」
- 5 秒冷却防止手机抖动时反复触发
- 新增 3 段音频素材

### v0.6.1-v0.6.2 — APK 体积优化（2026-07-06）
**里程碑**：Release APK 从 65 MB 压缩到 30 MB 出头。

**优化手段**：
- ABI 单架构：只打包 `arm64-v8a`（2019 年后所有 Android 设备均支持）
- 清理未引用的孤儿音频文件
- v0.7.0 继续启用 R8 代码压缩，再省 ~1.4 MB
- Debug APK 也同步优化：237 MB → 89 MB

**工程化**：新增单元测试断言所有 `assets/audio/*.mp3` 都被代码引用，防止误删在用音频或引入孤儿音频。

## 阶段三：无障碍红线与稳定性（2026-07-07，v0.7.x）

### 核心目标
真机测试暴露了严重的无障碍问题，推动了整个项目**最核心的设计原则**建立。

### v0.7.1 — P0 紧急修复：摄像头权限静默卡死 🔴
**触发事件**：v0.7.0 发布后用户反馈：首次安装拒绝摄像头权限后，APP 反复 log "相机不可用"，视障用户听不到任何声音，完全不知道发生了什么。

**根因分析**：
- `HomeScreen._initCameraWithRetry()` 把所有失败都当"硬件故障"
- 没有调用 `checkSelfPermission`，不知道当前权限状态
- 失败后只 `print` 了 4 个字就完事，**没有任何语音播报**

**这是一个转折点**：直接催生了项目最重要的设计原则 —— **NSF（No Silent Failure，视障工具绝不静默失败）**。

**NSF 红线内容**（写入 `AGENTS.md` §5.3，强制执行）：
> 视障用户无法通过屏幕看到任何错误信息。任何异常路径（权限拒绝、硬件不支持、初始化失败、网络中断、OCR 识别失败等）都必须有**语音播报**说明发生了什么、下一步该怎么办。
>
> **禁止替代方案**：
> - ❌ `print('error: $e')` 后无语音
> - ❌ `SnackBar` / `Toast`（视障用户看不到）
> - ❌ 仅靠 UI 颜色/文字变化
> - ❌ FileLogger 写日志后沉默（用户不知道有日志可看）
>
> **播报模板**：「出问题了」+「具体可操作的建议」

**v0.7.1 修复内容**：
- 新增 `PermissionService`，统一相机权限入口
- 区分「普通拒绝」和「永久拒绝」两种状态
- 永久拒绝时语音引导去系统设置，返回后自动重试
- 新增 3 段权限相关语音素材
- 14 个单元测试覆盖权限流程

### v0.7.2 — 弱光自动手电（2026-07-07）
**里程碑**：从"被动修复 bug"到"主动预判场景"，覆盖弱光环境下的使用体验。

**新增能力**：
- `LuminanceDetector`：用 dart:ui 内置能力解码 JPEG，BT.601 权重计算亮度，64 像素采样 < 5ms
- 亮度 < 40 判定为 dark，自动开启手电
- 手电硬件不支持时语音提示手动开启（NSF 原则）
- 8s 冷却防止每帧都播报，30s 冷却防止失败时反复 nag
- 生命周期 paused 时自动关手电（省电 + 防后台亮灯）
- 新增 3 段语音素材
- 19 个新单元测试

### v0.7.0 — R8 压缩与工程化（2026-07-07）
- R8 代码压缩启用，APK 再减 1.4 MB
- ProGuard 规则补全（Flutter 引擎、Kotlin 注解、Camera2 反射、ML Kit 等）
- 单元测试保护 R8 配置，防止未来误删
- `docs/TASKBOARD.md` 建立多 Agent 协作任务看板

## 阶段四：应用内更新与版本治理（2026-07-10 ~ 至今，v0.8.x）

### 核心目标
解决视障用户**无法手动升级**的致命问题，同时建立版本发布的治理机制。

### 问题背景
v0.7.x 修复了很多严重 bug，但视障用户怎么拿到新版本？
- 视障用户无法独立操作浏览器下载 APK
- 应用商店上架流程复杂，审核周期长
- 如果用户一直停留在有 bug 的旧版本，修复等于白做

**结论**：必须做**应用内自动更新**，让用户在 APP 内通过语音提示就能完成升级。

### v0.8.0 — 应用内更新机制（2026-07-10）
**里程碑**：用户无需手动下载，Wi-Fi 环境下自动检测更新，语音引导完成安装。

**技术方案**：
- 更新源策略：**Gitee 优先（国内快），GitHub 兜底**
- 检查时机：启动后、相机权限前（即使用户相机权限有问题，也能收到修复版本的更新提示）
- 节流策略：每周检查一次，仅 Wi-Fi 下检查（避免偷跑流量）
- 下载：Dio + 进度回调
- 安装：FileProvider 暴露缓存 APK + 系统安装器
- 权限处理：安装未知应用权限被拒时语音引导去设置，返回后自动重试

**新增内容**：
- 4 个新 Service：`UpdateService`、`DownloadService`、`InstallService`、`ConnectivityService`
- 9 段更新相关语音素材
- Android 权限新增：`INTERNET`、`REQUEST_INSTALL_PACKAGES`
- `android:allowBackup="false"`（避免备份导致的状态异常）
- 完整单元测试覆盖

### v0.8.1-v0.8.3 — Bug 修复与机制完善（2026-07-10 ~ 2026-07-11）
真机测试暴露了多个更新机制和权限判断的 bug：

**v0.8.1 修复**：
- **摄像头权限误判**（严重）：`shouldShowRequestPermissionRationale()` 在首次安装（从未请求过权限）时也返回 `false`，导致首次启动被误判为"永久拒绝"直接跳设置 → 用持久化标志区分"首次请求"和"永久拒绝"
- **应用名显示错误**：最小化时显示旧名"取餐助手" → 统一为"慧眼 SmartEye"
- **Dio 无超时**：网络不佳时启动阶段挂起 → 添加合理的超时设置

**v0.8.2 改进**：更新检查提前到权限检查之前

**v0.8.3 修复**（当前版本）：
- **权限标志备份问题**：SharedPreferences 被 ColorOS/Google 备份恢复，卸载重装后仍误判为永久拒绝 → 迁移到 `noBackupFilesDir`
- **安装权限返回无反应**：去设置开启权限后返回应用自动重试安装
- **APK 重复下载**：缓存已存在时跳过下载直接安装
- **下载超时过短**：连接超时从 10s 增加到 30s
- **GITEE_TOKEN 未持久化**：临时 export 关闭终端后丢失，导致发布时 Gitee 需要手动上传 → 写入 `.zshrc` 持久化，文档记录踩坑

### 版本治理机制建立（v0.8.3）
v1.0 是正式版里程碑，意味着从"公益原型"进入"可公开分发"阶段。为了防止仓促发布 v1.0，建立了**v1.0 发布冻结机制**：

- `VERSION.md` 列出 v1.0 readiness checklist（7 项条件：NSF 审计、5 款机型测试、文档同步、生产签名、用户走查、更新流程验证、人工批准）
- `scripts/release.sh` 硬拦截：未设置 `ALLOW_V1_RELEASE=1` 时禁止发布 major >= 1 的版本
- `AGENTS.md` 写入约束：未完成 checklist 前禁止提升 major 版本

### 双平台发布机制完善
- GitHub Release：通过 `gh` CLI 自动发布（keyring 持久化登录）
- Gitee Release：通过 `release-gitee.sh` 脚本 + `GITEE_TOKEN` 环境变量自动发布
- 一键发布：`./scripts/release.sh vX.Y.Z` 自动完成「测试→构建→推代码→打 tag→双平台 Release」全流程
- 发布 checklist 写入 `docs/IN_APP_UPDATE.md`

## 已知问题与技术债务

详见 `PROJECT.md` §2（已知问题）和 §3（技术债务），重点包括：

| 问题 | 级别 | 说明 |
|------|------|------|
| 音频叠加 | 🔴 高 | `speak()` 是 fire-and-forget，调用前必须先 `stop()`，禁止用 `delay` 等待 |
| 相机首次初始化超时 | 🟡 中 | ColorOS 冷启动可能慢，3 次自动重试 |
| 手写文字识别率低 | 🟡 中 | ML Kit 固有限制，仅支持印刷体（外卖小票是热敏打印，没问题） |
| flutter_assets 路径陷阱 | 🟡 中 | Kotlin 层必须加 `flutter_assets/` 前缀 |
| 调试日志覆盖层 | P1 | 正式版应隐藏或改为手势触发 |
| 双机种适配 | P3 | 目前仅 OPPO A96 测试过 |
