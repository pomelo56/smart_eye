# 版本管理规范

> 慧眼 SmartEye 采用语义化版本（Semantic Versioning）+ git tag 管理版本。

## 版本号格式

`MAJOR.MINOR.PATCH+BUILD`

| 字段 | 含义 | 何时递增 |
|------|------|---------|
| MAJOR | 不兼容的架构变更 | 大重构、技术栈更换 |
| MINOR | 新功能，向下兼容 | 新增平台、新交互、新语音 |
| PATCH | Bug 修复，向下兼容 | 修 OCR 误识别、修音频时序 |
| BUILD | 构建号（整数） | 每次发布递增 |

在以下 checklist **全部完成**之前，禁止将 `pubspec.yaml` 的 major 版本提升到 `1`，也禁止发布 `v1.0.0` 标签：

- [ ] 完成 5 款以上主流机型真机兼容性测试（含 OPPO / vivo / 小米 / 华为 / 荣耀）
- [ ] README.md / PRD.md / HANDOVER.md / TASKBOARD.md 与当前代码一致
- [ ] 生产签名配置验证通过，release APK 使用正式 keystore
- [ ] 用户无障碍走查通过（至少 2 位视障用户试用并反馈）
- [ ] 应用内更新、日志导出、权限流程在 Android 10+ 真机验证通过
- [ ] 核心维护者人工批准（不可由 AI 单独决策）

`scripts/release.sh` 已内置硬拦截：当 tag 或 `pubspec.yaml` 的 major >= 1 时，必须显式设置 `ALLOW_V1_RELEASE=1` 才能继续。

## 发布流程

每次迭代完成后：

1. 更新 `pubspec.yaml` 的 `version` 字段
2. 更新 `CHANGELOG.md`，将 `[Unreleased]` 归档为版本号
   - **严格遵循「版本描述写作规范」**（见 `CHANGELOG.md` 顶部）
   - 禁止口语化表达、占位符、随意批注
   - 每条变更必须写清楚「改了什么+为什么改+影响是什么」
   - 版本历史表格中的「说明」列必须在发布时填写完整，禁止留空或写「预留」
3. 更新本文件的「当前版本」和「版本历史」
4. 运行 `./scripts/test.sh` 确保全绿
5. 提交：`git commit -m "release: vX.Y.Z"`
6. 打标签：`git tag vX.Y.Z`
7. 推送：`git push && git push --tags`
8. 运行 `./scripts/release.sh vX.Y.Z` 自动发布双平台 Release（需提前配置 `gh` 和 `GITEE_TOKEN`）

## 回滚方法

```bash
# 查看所有版本
git tag -l

# 回滚到某个版本（保留工作区）
git checkout vX.Y.Z -- .

# 回滚到某个版本（完整重置）
git reset --hard vX.Y.Z
```
